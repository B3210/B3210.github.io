C语言答辩(1) 2021-11-22

C语言答辩(1)

项目1

设计思路

​ 之前做过什么扫描码的题，所以这个题直接想到了虚拟键值编码（所有系统上都是统一的），其他的也没想，就用这个方法实现了，也没多想其他方法。模拟按键有两个基本动作，即按下键和放开按键，对于两个值，直接接收用户敲键盘产生的数值就可以确定按的哪个键了。

可读性

降低与主函数的耦合度，提高函数内聚度。方便维护，体现模块化编程思想。

复杂度

控制流

​ 为了增加运行速度，牺牲了一部分可读性，用if语句代替switch语句,但是经实验证明，相差无几……

所以我把跳转功能加了switch语句，而键值判断用if语句合并成两个语句，都有用到可读的同时尽量简洁。（当然还有一个原因是switch只能接一个语句 Σ( ° △ °|||)︴）

健壮性

接收输入

​ 直接通过接收击键事件，读取统一的键盘键值，仅验证方向键键值，通过取非排除所有其他键。保证了游戏中敲击任意非法键，都会触发游戏结束。同时，也增加了用户体验。用户只需敲击方向键，而无需确认，敲击便触发验证，让这个程序更具有游戏的体验。

打印输出

在一些文本文件输出打印当中，有读取文件的判断，提醒读取错误，如打印历史成绩；且每次读写后都会关闭文件。

技术性

调参

srand(seed)，seed为随机数种子，为保证这是个反应类游戏，一系统时间（秒为单位），但这种刷新速度下，如果玩家操作很快，会通过一秒内反复按动相同方向键，快速刷分。对于更新种子太慢，我进行了调参使伪随机数更科学。

​ 之前我准备打印毫秒级别，但是经过反复实验验证，我发现

因此我把参数调低到一个合适速度，既不容易使程序崩溃，也容易出现短时间内不随机的情况，模型简化为（刷新时间等于或略大于随机数产生时间）

于是游戏变成了一定时间内反应得分，我以用户输入到存入内存这段时间作为反应时间

解耦

除了模块化函数以外，函数内也有解耦操作，如game( )函数中将键值存储为数组，以随机数产生索引值形式产生随机键值，保证随机数产生范围算法与键值独立开来，避免键值修改是算法实现复杂度增加，耦合度高移植性差。

内存管理

程序用到的变量很少，且为了减少内存开销，多采用指针，用动态内存（之前以为程序崩溃是内存原因，所以后面每次用完就把键值释放掉，“穷人的孩子会当家”︿(￣︶￣)︿）；还有一部分用全局变量减少函数内部为形参变量开辟内存空间；甚至连for语句都舍不得用嘞，都懒得定义循环变量。

创新性

如果以程序本身要实现的功能要求来看，那新添加的功能都算创新把……不过那些都是闹着玩的，有的确实是我“面向对象考虑的创新”

动态难度

通过等待用户输入时间来控制玩家反应时间，随着游戏分值提升，难度也会增加

其他什么再次游戏、成绩记录之类没什么好说的，体现不出灵感。(°ー°〃)

心得与收获

​ 总的来说，确实是体会到了“以赛促学”，学习了获取敲击键盘的事件并利用键值实现程序；也通过用户视角来考虑和调试程序（没有对象……所以没有面向对象经验![苦涩.jpg]）；最后也复习了一下指针，虽然用的不多（主要是没有用数据结构，太菜了Orz）。

Docker架构

Docker**(C/S)**架构程序，可以在同一台宿主机上运行 Docker 守护进程和客户端，也可以从本地的 Docker 客户端连接到运行在另一台宿主机上的远程 Docker 守护进程。

镜像与容器

(image)镜像是静态的定义，(container)容器是镜像运行时的实体。通过镜像启动一个容器，一个镜像就是一个可执行的包，其中包括运行应用程序所需要的所有内容：包含代码，运行时间，库，环境变量和配置文件等。

镜像分层

​ 通过扩展现有镜像，创建新的镜像。新镜像是从 base 镜像一层一层叠加生成的。每安装一个软件，就在现有镜像的基础上增加一层。

Docker Host 只需在磁盘上保存一份 base 镜像；同时内存中也只需加载一份 base 镜像，就可以为所有容器服务了。而且镜像的每一层都可以被共享。

Copy-on-Write

​ 容器层可写，镜像层只读。所有镜像层会联合在一起组成一个统一的文件系统（从上到下，会覆盖），容器层（新的可写层被加载到镜像的顶部）中，用户看到的是一个叠加之后的文件系统。容器层保存的是镜像变化的部分，不会对镜像本身进行任何修改。

volume数据卷

将宿主机的目录映射到容器中的目录，应用程序在容器中的目录读写数据会同步到宿主机上，这样容器产生的数据就可以持久化。

registry注册中心

Docker 用 Registry 来保存用户构建的镜像。Docker 公司提供了公共的镜像仓库，一个 Docker Registry 中可以包含多个仓库（Repository）；每个仓库可以包含多个标签（Tag）；每个标签对应一个镜像。

#<仓库名>:<标签> 的格式来指定具体是这个软件哪个版本的镜像。如果不给出标签，将以 latest 作为默认标签。
